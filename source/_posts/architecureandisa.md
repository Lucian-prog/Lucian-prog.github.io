## 预备概念：bit、byte、word 与地址

在进入正文前，需要先区分数据宽度、存储器地址和元素索引。后面讨论 `lw`、`sw` 和对齐时都会用到这些概念。

### bit：位

bit（比特、位）是最小的二进制信息单位，只能取 `0` 或 `1`。RTL 中的单比特信号就是一根逻辑线，例如：

```systemverilog
logic valid_i;
```

### byte：字节

byte（字节）固定由 8 bit 组成：

$$
1\ \text{byte}=8\ \text{bit}
$$

一个十六进制数字表示 4 bit，所以一个 byte 需要两个十六进制数字表示：

```text
8'b1010_0110 = 8'hA6
```

在 RISC-V 的字节寻址存储器中，每一个 byte 都有独立地址。地址增加 1，表示移动到相邻的下一个 byte。

### halfword：半字

在 RISC-V 中，halfword（半字）为 16 bit，也就是 2 byte：

$$
1\ \text{halfword}=16\ \text{bit}=2\ \text{byte}
$$

半字需要 4 个十六进制数字表示，例如 `16'h1234`。

### word：字

word（字）的宽度与具体体系结构有关，不能脱离上下文理解。在本书讨论的 RV32I 中：

$$
1\ \text{word}=32\ \text{bit}=4\ \text{byte}
$$

一个 32 位 word 需要 8 个十六进制数字表示，例如：

```text
32'h1234_ABCD
```

本文后面出现“字”时，若没有特别说明，都指 RV32I 中的 32 位 word。

### doubleword：双字

doubleword（双字）通常指 64 bit，也就是 8 byte：

$$
1\ \text{doubleword}=64\ \text{bit}=8\ \text{byte}
$$

RV32I 的基础整数寄存器宽度仍是 32 位。这里提前给出 doubleword，只是为了完整理解常见数据单位。

### 数据单位汇总

| 名称 | 英文       | 位数 | 字节数 | 十六进制数字数 |
| ---- | ---------- | ---: | -----: | -------------: |
| 位   | bit        |    1 |      - |              - |
| 字节 | byte       |    8 |      1 |              2 |
| 半字 | halfword   |   16 |      2 |              4 |
| 字   | word       |   32 |      4 |              8 |
| 双字 | doubleword |   64 |      8 |             16 |

### 字节地址、字起始地址与字索引

这三个概念不能混用：

- **字节地址**：存储器中某个 byte 的编号；
- **字起始地址**：一个 32 位 word 所占 4 个 byte 中，最低地址 byte 的地址；
- **字索引**：这是第几个 word，不是直接送给存储器的字节地址。

假设从地址 0 开始依次存放 32 位 word：

| 字索引 | 占用的字节地址 | 字起始地址 |
| -----: | -------------- | ---------: |
|      0 | 0、1、2、3     |          0 |
|      1 | 4、5、6、7     |          4 |
|      2 | 8、9、10、11   |          8 |
|      3 | 12、13、14、15 |         12 |

因此：

$$
\text{字起始地址}=\text{字索引}\times4
$$

例如“第 2 个 word”对应字索引 2，它的起始字节地址是 8。书中有时把这个起始字节地址简称为“字地址”，阅读时应根据上下文判断，不要误认为存储器改成了按字寻址。

### 一个 32 位 word 如何占用 4 个 byte

以数据 `32'h1234_ABCD` 为例，它可以拆成四个 byte：

```text
最高 byte                              最低 byte
  0x12          0x34          0xAB          0xCD
 data[31:24]   data[23:16]   data[15:8]    data[7:0]
```

这四个 byte 放到连续地址中的先后顺序由大小端决定；但无论采用大端还是小端，它始终占 4 个连续字节地址。数据宽度、地址和字节序是三个不同问题。

## 6.1 引言

### 1. 从数字电路上升到体系结构

书中给出的抽象层次大致是：

```text
应用软件
操作系统
体系结构
微体系结构
逻辑
数字电路
模拟电路
器件
物理
```

第四章和第五章主要位于“逻辑、数字电路、微体系结构构件”这一侧，第六章讨论的是体系结构（architecture）。

我的理解是：

- **体系结构**规定处理器对软件表现出的行为；
- **微体系结构**决定用什么硬件实现这些行为。

例如，体系结构规定 `add` 指令应当完成两个操作数相加并写回结果，但不会规定内部一定使用单周期、五级流水线还是多周期 ALU。

同一个 ISA 可以有很多种实现。只要它们执行同一程序时得到相同的体系结构结果，内部数据通路、流水级数、面积和频率都可以不同。

### 2. 指令、指令集与体系结构

一条指令（instruction）需要说明：

1. 执行什么操作；
2. 从哪里取得操作数；
3. 把结果放到哪里。

处理器能够执行的全部指令构成指令集（instruction set）。指令集、寄存器、数据类型、存储器组织和其他软件可见规则共同组成 ISA。

常见 ISA 有 RISC-V、ARM、x86、MIPS、SPARC 和 PowerPC。它们像不同的“语言”，具体语法和编码不同，但通常都会提供算术、逻辑、访存和控制转移等基本能力。

### 3. 汇编语言与机器语言

处理器硬件最终只能识别二进制编码，这种形式称为机器语言（machine language）。为了便于人阅读，使用助记符表示机器指令，这种形式称为汇编语言（assembly language）。

例如：

```asm
add s0, s1, s2
```

这里的 `add` 是操作助记符，`s1`、`s2` 是源操作数，`s0` 是目的操作数。汇编器会把它转换成对应的 32 位机器码。

从硬件角度看，这条指令最终意味着：

```text
寄存器堆读取 s1 和 s2
          ↓
ALU 执行加法
          ↓
结果写回 s0
```

### 4. 体系结构与微体系结构的区别

| 对比               | 体系结构                     | 微体系结构                         |
| ------------------ | ---------------------------- | ---------------------------------- |
| 关注内容           | 软件能看到什么               | 硬件内部怎样实现                   |
| 典型对象           | 指令、寄存器、地址、数据宽度 | ALU、寄存器堆、MUX、流水线、控制器 |
| 是否影响程序兼容性 | 会                           | 通常不会                           |
| 本书位置           | 第 6 章                      | 第 7 章                            |

这也是第六章和前面内容最重要的连接：

```text
第 4 章：会描述硬件
第 5 章：认识硬件构件
第 6 章：理解处理器必须完成的行为
第 7 章：把这些行为组织成处理器
```

### 5. 本章研究的 RV32I

书中主要讨论 RISC-V 32 位基础整数指令集 RV32I：

- `RV32`：基础整数寄存器和地址运算宽度为 32 位；
- `I`：基础整数指令集；
- 基础指令通常是 32 位；
- 处理 32 位整数数据；
- 使用 32 个通用整数寄存器；
- 存储器按字节寻址。

### 6. RISC-V 的四条设计原则

书中用四条设计原则贯穿对 RISC-V 的解释：

1. **规整性支持简单性**（regularity supports simplicity）；
2. **加速常用操作**（make the common case fast）；
3. **越小越快**（smaller is faster）；
4. **好的设计需要好的折中**（good design demands good compromises）。

从数字 IC 角度看，这四条原则会落到：

- 指令格式尽量规整，降低译码复杂度；
- 常用操作直接用简单指令完成；
- 寄存器堆不能无限增大，否则面积、延迟和布线压力上升；
- 指令宽度、立即数范围、寄存器数量和硬件复杂度需要相互折中。

## 6.2 汇编语言

### 6.2.1 概述

汇编语言是机器语言的可读表示。每条指令都要指出操作类型和操作数。

#### 1. 加法和减法

书中先从最简单的算术指令开始：

```c
a = b + c;
```

假设 `a`、`b`、`c` 分别保存在 `s0`、`s1`、`s2`：

```asm
add s0, s1, s2    # a = b + c
```

减法形式保持一致：

```c
a = b - c;
```

```asm
sub s0, s1, s2    # a = b - c
```

两条指令都是：一个目的寄存器、两个源寄存器。这种固定格式体现了“规整性支持简单性”。

硬件上可以直接对应寄存器堆的两个读端口和一个写端口：

```text
rs1 ─┐
     ├→ ALU → rd
rs2 ─┘
```

#### 2. 复杂表达式需要拆成多条指令

RISC-V 不用一条复杂指令完成任意表达式，而是把表达式拆成多个简单操作。例如：

```c
a = b + c - d;
```

可以写成：

```asm
add t0, s1, s2    # t0 = b + c
sub s0, t0, s3    # a  = t0 - d
```

这里 `t0` 保存中间结果。硬件每次仍然只执行一个简单 ALU 操作，复杂性由多条指令的组合承担。

#### 3. RISC 与 CISC

RISC-V 属于精简指令集计算机（RISC）：

- 指令数量和格式相对规整；
- 单条指令完成较简单的操作；
- 复杂任务由多条指令组合；
- 译码和数据通路更容易保持规则。

x86 常被归为复杂指令集计算机（CISC），包含更多复杂操作和变长编码。现代处理器的实际实现已经很复杂，不能仅凭“RISC/CISC”判断性能，但对初学处理器 RTL 来说，RV32I 的规整性确实更适合从译码器和数据通路开始实现。

### 6.2.2 操作数：寄存器、常数和内存

处理器的数据主要放在两个地方：

- 寄存器：容量小、速度快；
- 存储器：容量大、访问相对慢。

立即数则直接编码在指令中，不需要先从寄存器或存储器取出。

#### 1. 寄存器操作数

RV32I 有 32 个整数寄存器，编号为 `x0`～`x31`。汇编中通常使用 ABI 别名，例如 `s0`、`t0`、`a0`。

| 寄存器      | 编号         | 书中用途          |
| ----------- | ------------ | ----------------- |
| `zero`      | `x0`         | 常数 0            |
| `ra`        | `x1`         | 返回地址          |
| `sp`        | `x2`         | 栈指针            |
| `gp`        | `x3`         | 全局指针          |
| `tp`        | `x4`         | 线程指针          |
| `t0`～`t2`  | `x5`～`x7`   | 临时寄存器        |
| `s0/fp`     | `x8`         | 保存寄存器/帧指针 |
| `s1`        | `x9`         | 保存寄存器        |
| `a0`～`a1`  | `x10`～`x11` | 参数/返回值       |
| `a2`～`a7`  | `x12`～`x17` | 参数              |
| `s2`～`s11` | `x18`～`x27` | 保存寄存器        |
| `t3`～`t6`  | `x28`～`x31` | 临时寄存器        |

这篇先记名称和编号范围，不展开调用者保存、被调用者保存和栈帧。那些内容在 `6.3.7 函数调用` 再学更自然。

##### 寄存器堆的硬件理解

32 个寄存器意味着寄存器地址需要 5 位：
$$
2^5=32
$$

R-type 算术指令通常需要同时读两个源寄存器、写一个目的寄存器，所以寄存器堆一般设计为：

```text
2 个组合读端口 + 1 个时序写端口
```

`x0` 是特殊寄存器：

- 读取 `x0` 必须得到 0；
- 对 `x0` 的写入必须被忽略。

RTL 中通常通过“读地址为 0 时输出 0”和“写地址不为 0 时才允许写入”实现，而不是依赖复位后 `regs[0]` 恰好为 0。

#### 2. 常数/立即数操作数

立即数（immediate）直接包含在指令中。例如：

```asm
addi s0, s0, 4     # s0 = s0 + 4
addi s1, s0, -12   # s1 = s0 - 12
```

书中这一部分使用的 `addi` 立即数是 12 位补码，范围为：

$$
-2^{11}\leq imm\leq 2^{11}-1
$$

即 `-2048`～`2047`。进入 32 位 ALU 前，需要进行符号扩展：

```text
12'hFFC
符号扩展后 = 32'hFFFF_FFFC
```

##### 立即数数据通路的硬件理解

`addi` 与 `add` 可以复用同一个加法器，区别主要在 ALU 的第二个输入：

```text
add : ALU_B = 寄存器 rs2
addi: ALU_B = 符号扩展后的立即数
```

因此数据通路需要一个多路选择器，在寄存器数据和立即数之间选择。

立即数符号扩展是组合逻辑，综合后主要是连线和符号位复制，不会综合成寄存器。

#### 3. 大常数与 `lui`

12 位立即数无法表示任意 32 位常数。书中使用 `lui` 把 20 位立即数写入寄存器高 20 位，低 12 位补 0，再用 `addi` 补低位。

例如：

```asm
lui  s2, 0xABCDE      # s2 = 0xABCDE000
addi s2, s2, 0x123    # s2 = 0xABCDE123
```

需要注意：如果目标常数低 12 位的 bit 11 为 1，`addi` 会把它解释为负数。这时传给 `lui` 的高 20 位通常需要加 1。

例如目标值 `0xFEEDA987`：

```asm
lui  s2, 0xFEEDB      # s2 = 0xFEEDB000
addi s2, s2, -1657    # 12 位编码为 0x987
                      # s2 = 0xFEEDA987
```

##### `0xFEEDA987` 的具体拆分计算

先观察目标值的低 12 位：

```text
目标值：0xFEEDA987
高 20 位：0xFEEDA
低 12 位：0x987
```

`0x987` 写成 12 位二进制是：

```text
12'h987 = 12'b1001_1000_0111
              ↑
           bit 11 = 1
```

由于最高位 bit 11 为 1，它作为 12 位补码时表示负数。先把 `0x987` 当作无符号数计算：

```text
0x987 = 9×256 + 8×16 + 7
      = 2439
```

12 位补码的模为 $2^{12}=4096$，因此其有符号值为：

$$
2439-4096=-1657
$$

所以汇编中写 `-1657`，指令的 12 位立即数字段实际编码为 `0x987`。送入 32 位 ALU 前，它会进行符号扩展：

```text
12 位立即数： 0x987
符号扩展后：  0xFFFFF987
```

如果先执行 `lui s2,0xFEEDA`，得到 `0xFEEDA000`，再加上符号扩展后的负数，结果会少 `0x1000`：

```text
0xFEEDA000 - 0x679 = 0xFEED9987    // 错误结果
```

其中 `0x679=1657`。因此需要预先把 `lui` 的高 20 位从 `0xFEEDA` 加 1 调整为 `0xFEEDB`：

```text
  0xFEEDB000
- 0x00000679
--------------
  0xFEEDA987
```

用 32 位补码加法表示同一个过程：

```text
  0xFEEDB000
+ 0xFFFFF987
--------------
  0x1FEEDA987
```

32 位寄存器只保留低 32 位，最终结果就是：

```text
s2 = 0xFEEDA987
```

可以把大常数的拆分记成：

```text
hi20 = (value + 0x800) >> 12
lo12 = value[11:0] 按 12 位补码解释
```

本例中：

```text
hi20 = (0xFEEDA987 + 0x800) >> 12 = 0xFEEDB
lo12 = 12'h987 = -1657
```

这里不是加法器出错，而是 12 位立即数符号扩展后的正常结果。高 20 位加 1，是为了补偿低 12 位被解释为负数造成的借位。

#### 4. 内存操作数

寄存器只有 32 个，不可能保存程序中的全部数据，因此还需要内存。RV32I 是 Load/Store 体系结构：

- ALU 指令主要操作寄存器；
- 内存数据必须先通过 Load 读入寄存器；
- 运算结果要通过 Store 才能写回内存。

这意味着基础数据通路中，存储器不会直接作为普通 `add` 的一个操作数。

#### 5. 字节寻址

RV32I 存储器按字节寻址，每个字节有独立地址。一个 32 位字占 4 个字节：

```text
字 0 的起始地址：0
字 1 的起始地址：4
字 2 的起始地址：8
字 3 的起始地址：12
```

因此：

$$
word\_address=word\_index\times4
$$

这里容易混淆的是：`lw` 和 `sw` 的偏移量单位是字节，不是“第几个字”。

#### 6. `lw`：从内存读一个字

书中示例：

```c
a = mem[2];
```

若 `mem` 从地址 0 开始，`mem[2]` 是第 2 个 32 位字，其字节地址为 `2×4=8`：

```asm
lw s7, 8(zero)    # s7 = Mem32[0 + 8]
```

`lw` 使用“基址寄存器 + 立即数偏移”计算有效地址：
$$
effective\_address=rs1+sext(imm12)
$$

##### `lw` 的硬件信号流

```text
基址寄存器  ─────────┐
                   ├→ ALU 加法 → 数据存储器地址
符号扩展立即数 ──────┘

数据存储器读数据 → 写回寄存器堆
```

`lw` 需要：

1. 读取基址寄存器；
2. 生成并符号扩展立即数；
3. 用 ALU 计算地址；
4. 读取数据存储器；
5. 把数据写回目的寄存器。

这也是后续单周期处理器中比较长的一条组合路径。

#### 7. `sw`：向内存写一个字

书中示例：

```c
mem[5] = 42;
```

第 5 个字的字节地址为 `5×4=20`：

```asm
addi t3, zero, 42    # t3 = 42
sw   t3, 20(zero)   # Mem32[20] = t3
```

##### `sw` 的硬件信号流

```text
基址寄存器 + 立即数 → ALU → 数据存储器地址
源寄存器 t3 ─────────────→ 数据存储器写数据
```

`sw` 与 `lw` 都要使用 ALU 计算地址，但二者的状态更新不同：

| 指令 | 读内存 | 写内存 | 写寄存器 |
| ---- | ------ | ------ | -------- |
| `lw` | 是     | 否     | 是       |
| `sw` | 否     | 是     | 否       |

后续写控制器时，`sw` 必须关闭寄存器写使能并打开存储器写使能；`lw` 则相反。

#### 8. 对齐问题

##### 什么是对齐

若一个数据对象的起始地址是其字节数的整数倍，就称为自然对齐（naturally aligned）。统一写成：

$$
address\bmod size=0
$$

其中：

- `address` 是数据的起始字节地址；
- `size` 是数据占用的字节数。

不同宽度的自然对齐条件如下：

| 访问宽度   | 字节数 | 自然对齐条件    | 地址低位条件          |
| ---------- | -----: | --------------- | --------------------- |
| byte       |      1 | 地址是 1 的倍数 | 无限制                |
| halfword   |      2 | 地址是 2 的倍数 | `address[0]=0`        |
| word       |      4 | 地址是 4 的倍数 | `address[1:0]=2'b00`  |
| doubleword |      8 | 地址是 8 的倍数 | `address[2:0]=3'b000` |

byte 只占 1 个字节，所以任意地址都满足 byte 对齐。32 位 word 占 4 个字节，因此 `lw/sw` 的自然对齐地址应为 4 的倍数：

```text
对齐地址：   0x1000、0x1004、0x1008、0x100C
未对齐地址： 0x1001、0x1002、0x1003、0x1005
```

##### 为什么未对齐访问可能跨界

假设数据存储器每次读取一个对齐的 32 位 word：

```text
对齐存储字 0：地址 0x1000～0x1003
对齐存储字 1：地址 0x1004～0x1007
```

执行对齐访问：

```asm
lw t0, 0x1000(zero)
```

需要的 4 个 byte 全部位于第一个存储字中：

```text
0x1000、0x1001、0x1002、0x1003
```

一次 32 位存储器读取即可完成。

如果从 `0x1002` 开始读取 32 位 word：

```text
需要的地址：0x1002、0x1003、0x1004、0x1005
```

它跨越了两个对齐存储字：

```text
第一个存储字提供：0x1002、0x1003
第二个存储字提供：0x1004、0x1005
```

硬件若要支持这次未对齐 `lw`，通常需要：

1. 读取第一个对齐 word；
2. 读取第二个对齐 word；
3. 根据地址低位分别移位；
4. 从两个结果中拼接出最终 32 位数据；
5. 保证整个操作对软件表现为一条指令。

未对齐 `sw` 更复杂，因为它可能需要对两个存储字分别产生写使能，并保证不能只成功写入一半。

##### 地址在 32 位存储器中的拆分

若底层 RAM 以 32 位 word 为一个存储单元，字节地址可以拆为：

```text
address[31:2]：选择第几个 32 位存储字
address[1:0] ：选择该存储字内部的第几个 byte
```

对应关系为：

```text
word_index  = byte_address >> 2
byte_offset = byte_address[1:0]
```

例如字节地址 `0x1006`：

```text
word_index  = 0x1006 >> 2 = 0x401
对齐字基址 = 0x401 << 2 = 0x1004
byte_offset = 2
```

因此它选择的是起始地址为 `0x1004` 的 32 位存储字中的第 2 个 byte。

## 本篇硬件视角总结

本篇按照书中 `6.1 → 6.2` 的顺序，先认识体系结构，再认识汇编和三类操作数。

### 目前出现的硬件资源

| 书中概念       | 对应硬件                    |
| -------------- | --------------------------- |
| 32 个寄存器    | 32×32 位寄存器堆            |
| 两个源寄存器   | 两个寄存器读端口            |

| `x0` 恒为 0    | 寄存器堆读写保护逻辑        |

### 与第四章的连接

- 寄存器堆写入属于时序逻辑，使用非阻塞赋值；
- 读取寄存器、立即数扩展、ALU 和地址计算属于组合逻辑；
- 组合译码必须有安全默认值，避免锁存器和错误写使能；
- 目前只是从 ISA 语义推导硬件需求，完整 RTL 要等机器码格式和数据通路学完后再写。
